## 2. Root cause investigation

- CDP 연결과 target 탐지는 정상이어도 state에는 이전 DOM 값이 `healthy/fresh`로 남을 수 있었다.
- 기존 success 판정은 `parseResult.ok`에 과하게 의존했다.
- source URL exact match, provider URL guard, expected usage label, reload-read 여부가 fresh 판정의 필수 조건으로 강제되지 않았다.
- Swift 메뉴 새로고침은 `monitor` 단발 실행만 호출했고, 앱 소유 monitor loop가 lock을 잡고 있으면 단발 실행이 skip될 수 있었다.

## 3. Regression cause

- V3 구조에서 backend diagnostics와 state freshness metadata가 분리되면서 “parser success”와 “verified live read success”가 같은 의미처럼 처리됐다.
- `refresh-now` command와 menu refresh가 reload-read 경로로 완전히 연결되지 않았다.
- status/health에 freshness 근거가 부족해 stale state가 fresh처럼 보이는 문제를 빠르게 확인하기 어려웠다.

## 4. Fix summary

- CDP success를 `parseResult.ok` 단독이 아니라 freshness validation 결과로 확정하게 변경했다.
- Codex 기본 URL을 `https://chatgpt.com/codex/cloud/settings/analytics`로 정리했다.
- selected target metadata에 exact URL match, URL guard, candidate count, expected label, reload-read 여부를 기록했다.
- `verify:usage-source`가 reload-read, parser output, normalized output, state before/after, status-json comparison을 출력하고 state를 실제 갱신하게 했다.
- Swift menu refresh는 `--force-reload`를 사용하고, stale이면 gauge 값을 fresh 값처럼 표시하지 않게 했다.
- `--force-reload` 단발 실행은 기존 앱 monitor loop lock이 있어도 1회 reload-read cycle을 수행하게 했다.
- package version을 `3.0.1-rc.1`로 올렸다.

## 5. Freshness decision policy

`fresh`는 아래 조건을 통과할 때만 저장한다.

- selected target 존재
- selected URL이 provider URL guard 통과
- exact configured URL match 또는 provider URL match 근거 존재
- extraction error 없음
- expected usage label 존재
- parser valid
- remaining short/weekly normalization valid
- force reload/verify/manual refresh이면 reload 이후 read 성공

실패하면 previous usage는 보존하고 `freshness`는 `stale` 또는 `unknown`으로 저장한다.

## 6. Source selection policy

- exact configured URL match가 최우선이다.
- Codex expected URL: `https://chatgpt.com/codex/cloud/settings/analytics`
- Claude expected URL: `https://claude.ai/settings/usage`
- title만 보고 target을 선택하지 않는다.
- provider URL pattern은 exact match가 없을 때만 fallback 근거로 사용한다.

## 7. Reload-read policy

- `npm run verify:usage-source`: 항상 reload-read.
- Swift menu refresh: `npm run monitor -- --dry-run-notifications --force-reload`.
- Diagnostics `refresh-now`: 모든 provider force reload.
- Diagnostics `reload-source`: 해당 provider force reload.
- regular polling은 기존 retry/cooldown 정책을 유지한다.

## 8. Parser/scenario changes

- Codex fixture 추가: `5시간 사용 한도 0% 남음`, `주간 사용 한도 31% 남음` → remaining `0/31`.
- Claude fixture 추가: `현재 세션 81% 사용됨`, `모든 모델 14% 사용됨` → remaining `19/86`.
- Claude parser는 `현재 세션`과 `모든 모델` anchor를 더 강하게 반영한다.
- URL mismatch parser success는 fresh가 되지 않는 scenario를 추가했다.
- expected label 없는 parser success는 fresh가 되지 않는 scenario를 추가했다.
- failed read가 previous usage와 `lastFreshReadAt`을 보존하는 scenario를 강화했다.

## 9. State update changes

- success 시 기록: usage, `status=healthy`, `freshness=fresh`, `lastFreshReadAt`, `lastSuccessfulCheckedAt`, target/source metadata, freshness decision reason.
- failure 시 기록: `lastAttemptAt`, `lastParseFailedAt`, `lastFailureAt`, `lastError`, `consecutiveFailures`, stale/unknown freshness.
- failure 시 보존: previous usage, `lastFreshReadAt`, `lastSuccessfulCheckedAt`.
- failure 시 금지: 100/0 기본값 덮어쓰기, fresh timestamp 갱신.

## 12. Screenshot comparison

- 사용자가 제공한 초기 screenshot 기준 값은 작업 중 live usage 변화로 더 이상 고정 재현되지 않았다.
- 최종 비교는 reload 직후 safe context와 parser/status-json 값을 기준으로 수행했다.
- 최종 live page safe context 기준:
  - Codex: `71% 남음`, `26% 남음`
  - Claude: `0% 사용됨`, `14% 사용됨` → remaining `100/86`
- full screenshot 파일은 개인정보와 페이지 주변 텍스트 노출 위험 때문에 report에 저장하지 않았다.

## 13. Remaining risks

- launchd plist는 현재 설치/로드되어 있지 않아 health는 launchd 항목을 error로 표시한다.
- live usage 값은 검증 중에도 계속 변할 수 있다.
- CDP target이 동일 URL 여러 개일 때는 exact URL 후보 중 뒤쪽 tab 우선 정책을 유지한다.
- full visual menu screenshot 검증은 저장하지 않았고, Swift compile/test와 status-json decode 경로로 확인했다.
